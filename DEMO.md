# 退票候补递补服务 - 演示指南

## 快速启动

### 方式一：Docker 启动（推荐）

```bash
docker-compose up -d --build
```

启动后访问：http://localhost:3000

首次启动会自动初始化演示数据。

### 方式二：本地启动

```bash
# 1. 安装后端依赖并初始化数据
cd backend
npm install
node seed.js

# 2. 启动后端服务
node server.js

# 3. 安装前端依赖并启动（另开一个终端）
cd ../frontend
npm install
npm run dev
```

前端访问：http://localhost:5173
后端 API：http://localhost:3000

## 完整演示路径

### 第一步：查看初始状态

1. 打开管理后台，点击左侧「演出管理」
2. 点击第一场「周杰伦演唱会」进入详情
3. 查看**票档概览**：
   - 内场VIP：2000元，无可用票，2人候补
   - 内场A：1580元，有 2 张可用票（已被锁定），4人候补
   - 看台一等：980元，有 1 张可用票（已被锁定），2人候补
   - 看台二等：580元，有 3 张可用票（已被锁定），1人候补
   - 看台三等：380元，无可用票，1人候补

4. 切换到**候补队列**标签页，查看所有候补用户的状态
5. 切换到**锁定记录**标签页，查看当前有哪些用户被锁定了名额

### 第二步：查看递补日志，理解为什么是这些人拿到了票

1. 切换到**递补日志**标签页
2. 点击最新一条「系统初始化」日志的「查看详情」
3. 可以看到完整的分配过程：
   - 每个人的排名
   - 为什么命中（分配成功）
   - 为什么跳过（票档不匹配？连座不满足？名额被占？）

**以「内场A」票档为例：**
- 郑十（排名第1）：需要 2 连座 → ✓ 分配成功（找到 2 个连座）
- 孙七（排名第2）：需要 4 连座 → ✗ 跳过（剩余票数不足）
- 王五（排名第3）：需要 3 连座 → ✗ 跳过（剩余票数不足）
- 李四（排名第4）：需要 1 座 → ✗ 跳过（无可用座位）

### 第三步：体验跨场次名额占用

**张三**同时在两场演唱会的候补队列中：
- 北京站（周杰伦）：内场VIP，2连座
- 上海站（五月天）：内场VIP，1座

由于张三已经被上海站锁定了名额，所以即使在北京站排第一也拿不到票。

1. 点击左侧「用户管理」
2. 找到「张三」，点击「查看详情」
3. 可以看到张三当前在上海站有一个锁定名额
4. 回到北京站详情，查看「内场VIP」的递补日志
5. 可以看到张三被跳过的原因：「用户已在另一演出锁定名额」

### 第四步：模拟退票，观察自动递补

1. 在周杰伦演唱会详情页，切换到**座位管理**标签
2. 找到一个状态为「已售出」的座位（比如 A1 排 1 号）
3. 点击「退票」按钮，确认退票
4. 系统会自动触发递补分配
5. 切换到**递补日志**标签，查看最新的一条「退票触发」日志
6. 点击「查看详情」，可以看到：
   - 退的票分配给了谁
   - 跳过了哪些人，分别是什么原因

**可以尝试退一张「内场A」的票：**
- 退一张散座
- 观察：需要 3 连座、4 连座的人会被跳过（不降级）
- 只接受单座的人会优先获得

### 第五步：用户确认购票

1. 切换到**锁定记录**标签
2. 找到一条状态为「锁定中」的记录
3. 点击「确认」按钮，模拟用户确认购票
4. 确认后，该名额从「锁定中」变为「已确认」
5. 切换到**已确认订单**标签，可以看到这条记录

**重要规则：已确认的票不会因为后面有人排得更早而被改动。**

### 第六步：体验超时释放

锁定名额默认有效期为 5 分钟。系统每 10 秒自动检查过期锁定。

1. 在「锁定记录」中找到一个即将过期的锁定
2. 等待 5 分钟（或者用「释放」按钮手动模拟超时）
3. 超时后，名额会自动释放
4. 系统会自动重新递补给下一位候补用户
5. 查看最新的递补日志，观察名额的流转

### 第七步：候补取消触发重新分配

1. 在**候补队列**中找到一个状态为「等待中」的用户
2. 点击「取消候补」
3. 如果取消的是排在前面的用户，后面的人不会受影响
4. 但如果取消的是已锁定（待确认）的用户，释放的名额会重新分配
5. 查看递补日志，观察取消候补触发的重新分配

### 第八步：手动触发递补

如果有新的退票或候补取消，系统会自动触发递补。
运营人员也可以手动触发：

1. 在**票档概览**中，点击某个票档的「手动触发递补」按钮
2. 系统会重新运行一次分配算法
3. 查看最新的递补日志

## 核心规则总结

### 分配优先级
1. **候补提交时间**：越早提交，排名越靠前
2. **票档匹配**：只考虑同票档的候补
3. **连座需求**：
   - 用户只接受连座，找不到连座就跳过，**不降级为散座**
   - 退出来的是多张连座票，优先分配给连座需求匹配的用户
4. **跨场次占用限制**：
   - 同一用户同时只能在一场演出中有锁定名额
   - 如果在 A 场锁定了，B 场有新退票也不会递给他
   - 等 A 场确认成功或超时释放后，B 场才能递补给他

### 状态流转
```
候补等待 → 锁定待确认 → 确认成功
                ↓
             超时释放 → 回到候补 → 重新分配
                ↓
          用户主动放弃 → 回到候补 → 重新分配
```

### 已确认保护
- 已经确认成功的订单，不会因为后面有人排得更早而被取消
- 只有锁定中的名额才会参与重新分配

## API 接口（快速测试用）

```bash
# 获取演出列表
curl http://localhost:3000/api/shows

# 获取某演出的候补队列
curl http://localhost:3000/api/shows/show_001/waitlist

# 获取某演出的递补日志
curl http://localhost:3000/api/shows/show_001/allocation-logs

# 查看递补日志详情
curl http://localhost:3000/api/allocation-logs/{logId}

# 退票（触发递补）
curl -X POST http://localhost:3000/api/seats/{seatId}/refund

# 确认锁定
curl -X POST http://localhost:3000/api/locks/{lockId}/confirm

# 释放锁定
curl -X POST http://localhost:3000/api/locks/{lockId}/release
```
